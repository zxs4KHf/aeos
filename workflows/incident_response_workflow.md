# 🚨 故障排查与复盘工作流 (Incident Response Workflow) v0.6.0

当发生系统异常死锁、任务静默丢包、外部第三方服务故障或代码部署崩溃时，AI 智能体必须执行本工作流，以确保故障被快速控制、彻底根治并在复盘中固化防御防线。

---

## 故障响应生命周期

```text
[报警/捕获 (Incident Detected)]
               │
               ▼
[1. 紧急避险与防御 (Mitigation)] ──► 执行版本回退或热修复 (Rollback/Hotfix)
               │
               ▼
[2. 根本原因定位 (RCA)] ──────────► 还原现场，通过 5 Whys RCA 刨析到设计层
               │
               ▼
[3. 彻底根治编码 (Remediation)] ────► 编写排他锁自愈或逻辑重构代码
               │
               ▼
[4. 复盘文档归档 (Postmortem)] ────► 编写 Blameless Postmortem 并固化行动项
```

---

## 一、 报警与捕获阶段 (Detection)
1. **自动状态监听**：
   - 智能体应通过后台 Poller 日志或数据库状态，定期巡检任务执行延迟。
   - 若检测到某一任务处于 `started` 状态超过 10 分钟无进度，或 Poller 频繁退出，应立即判定为故障事件（Incident）。
2. **静默故障判定**：
   - 若接收到用户关于“下发任务没反应/Bot 没回话”的反馈，智能体应优先检查全局进程状态及 SQLite 锁状态，禁止盲目猜测。

---

## 二、 紧急避险与防御阶段 (Mitigation)
> 首要目标是恢复服务，而不是写完美代码。必须在 5 分钟内完成服务止血。

1. **版本紧急回退 (Rollback)**：
   - 若因最近的 Git 提交导致崩溃，立即运行 `git rollback` 或 `git revert` 到上一个稳定的语义化 Tag 版本。
2. **清理僵尸进程与锁文件**：
   - 若数据库死锁，运行进程检查工具，强行杀死持锁不放的孤儿 Node 进程（如残留的旧 `poll_and_wait.js`），并手动删除 `.lock` 状态文件。
3. **降级离线模式 (Fallback)**：
   - 若第三方 API（如飞书服务）抖动，立即启动适配器层预留的 Mock 数据模式，保证本地任务管道不抛错崩溃。

---

## 三、 根本原因定位阶段 (RCA)
> 在服务恢复后，必须立刻转入 RCA 分析。

1. **五问分析法 (5 Whys)**：
   - 智能体必须连续追问五次“为什么”，直到追查到系统底层设计缺陷或流程机制缺失，禁止停留在“用户操作失误”或“外部网络抖动”等浅表层原因。
2. **日志追溯与报错定位**：
   - 提取并分析 `logs/` 下的报错 stack，找出导致进程异常退出的具体文件及行数。

---

## 四、 彻底根治阶段 (Remediation)
1. **架构性重构修复**：
   - 针对 RCA 暴露的瓶颈进行编码修复。例如：重构 poller 引入 `process.ppid` 定时存活检测，彻底从机制上解决孤儿进程问题。
2. **编写防复现测试用例**：
   - 针对该故障的触发边界，在 `tests/` 下编写专门的单元测试用例，确保后续的代码迭代中该问题不会复现（Regression Testing）。

---

## 五、 复盘与固化阶段 (Postmortem)
1. **起草 Blameless Postmortem**：
   - 依照 [事故复盘模板](file:///d:/vibecoding/aeos/templates/postmortem_template.md) 撰写报告，客观记录时间线与根本原因。
2. **制定 Action Items（行动计划）**：
   - 列出具体的后续防御开发任务（如：增加监控报警、重构底层锁接口），明确分工与截止时间。
3. **流程固化与归档**：
   - 将故障排查的踩坑点同步写入 `memory/LESSONS_LEARNED.md`，并将 Postmortem 文档通过 Git 提交至代码库中。
