# 评审与反馈同步工作流 v2.0

Status: active

本工作流定义与平台无关的评审协议。侧边面板、Artifact、PR 评论或本地 Markdown 都只是可选媒介，AEOS 不假定它们一定存在。

## 一、确定权威来源

开始评审前明确：

- 哪个仓库文件或提交是当前 source of truth。
- 反馈通过什么媒介提交。
- 谁负责接受、拒绝和合并反馈。
- 是否存在并发编辑者。

镜像或导出文件必须标明来源版本，不能在没有版本信息时自动覆盖原文件。

## 二、提交评审材料

评审摘要应包含：目标、非目标、关键设计选择、主要差异、验证证据和已知风险。对小型变更可直接使用代码 diff 和交付摘要，不要求额外 Walkthrough 文件。

## 三、处理反馈

1. 读取最新反馈并确认其对应的源版本。
2. 将反馈分类为接受、拒绝、需澄清或超出范围。
3. 应用接受项前重新检查本地文件，避免覆盖并发修改。
4. 发生冲突时保留双方内容，展示最小冲突范围并请求决策。
5. 修改后重新运行受影响验证，并报告反馈处理结果。

## 四、平台适配

- 原生评审 UI：使用宿主提供的评论、diff 和确认能力。
- Pull Request：将讨论与具体行或提交绑定。
- 文件评审：使用版本化 Markdown，并记录基线 commit 或内容哈希。
- 纯对话：在回复中给出紧凑摘要和明确问题。

任何媒介都不得被解释为自动授权提交、推送、部署或覆盖用户文件。这些操作保持独立授权。
