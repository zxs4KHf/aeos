# AEOS 测试与验证标准 v2.0

Status: active

## 一、按风险设计测试

- 纯逻辑和边界条件优先使用快速单元测试。
- 文件系统、数据库、队列和外部协议的真实集成行为使用隔离的集成测试。
- 关键用户流程、部署配置和跨服务合同使用少量端到端或冒烟测试。
- 不要求所有外部调用都 Mock；测试目标决定使用 fake、stub、sandbox 还是真实依赖。

## 二、变更验证

1. 从最接近改动的可靠检查开始。
2. 公共接口、共享库、数据结构或用户主路径变化时扩大回归范围。
3. 修复 Bug 时优先加入能证明原问题的回归测试。
4. 测试失败不能通过删除断言或降低门槛掩盖；行为确需变化时同步更新规格和测试理由。

## 三、覆盖率

覆盖率是发现盲区的信号，不是通用质量目标。项目可以设置门槛，但应结合分支风险、变异测试、缺陷历史和关键路径，而不是机械套用统一百分比。

## 四、测试质量

- 测试结果应确定、可重复并隔离自身状态。
- 断言关注外部可观察行为，避免过度绑定实现细节。
- 慢测试、易抖动测试和生产依赖需要显式标记和稳定策略。
- 交付时报告实际运行的检查和未运行原因。
