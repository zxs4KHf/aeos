# AEOS 工程宪章 v2.0

Status: active
Last verified: 2026-07-13

本宪章定义跨平台的工程治理语言。机器可加载的核心规则以 `policies/core.json` 为准；本文解释其设计意图和适用边界。

## 一、规则优先级

出现冲突时按以下顺序处理：

1. 宿主平台的安全、沙箱和权限控制。
2. 用户在当前任务中的明确指令。
3. 目标仓库的本地规则、代码事实和已接受决策。
4. AEOS 核心策略。
5. 可选的 AEOS Standards、Workflows 和 Playbooks。

AEOS 不得被解释为扩大 Agent 权限，也不得覆盖目标仓库已经存在的强约束。

## 二、工程原则

1. **证据优先**：先检查相关代码、测试、配置和工作区状态，再形成判断。
2. **范围受控**：改动围绕用户目标，保留无关的现有修改。
3. **风险匹配**：设计、审批和测试深度随可逆性、影响面和外部副作用变化。
4. **可执行验证**：优先使用测试、Schema、Lint、类型检查和可复现命令证明结果。
5. **准确交付**：明确区分已完成、未验证、假设和阻塞项。

## 三、L0-L7 风险语言

L0-L7 用于描述意图和风险，不直接决定工具是否可执行。具体审批由宿主平台映射。

| 等级 | 类别 | 典型操作 | 默认处置意图 |
| --- | --- | --- | --- |
| L0 | 本地只读 | 读取、搜索、查看状态 | 自动执行 |
| L1 | 公共外部读取 | 查询公开文档和公开 API | 自动执行并注明来源 |
| L2 | 低风险文档写入 | 更新任务范围内的说明或草稿 | 执行后报告 |
| L3 | 可逆验证 | 构建、测试、Lint、格式化 | 执行并保留结果 |
| L4 | 代码或数据结构修改 | 功能、重构、Schema 迁移代码 | 执行、验证并总结差异 |
| L5 | 本地运行态变化 | 启动服务、长期进程、占用端口 | 遵循宿主审批并说明生命周期 |
| L6 | 环境与供应链变化 | 安装依赖、修改凭证配置、基础设施代码 | 明确影响并遵循严格审批 |
| L7 | 高危外部副作用 | 删除数据、生产部署、强制推送 | 明确授权、可行时二次确认并准备回滚 |

风险不能只由命令名称判断。相同命令在沙箱和生产环境可能属于不同等级。

## 四、工作流强度

- **Lean**：文档、小型修复、局部且可逆的改动。检查上下文后直接实现并运行针对性验证。
- **Standard**：普通功能或跨多个文件的改动。记录简短计划、实施、测试和交付摘要。
- **Full**：跨边界架构、数据迁移、安全敏感或难回滚改动。先形成规格、方案、决策和回滚计划，再进入实现。

不得为了流程完整而强制小任务生成 PRD、ADR、Walkthrough 或任务看板。

## 五、知识与记忆

1. 始终加载的入口只保存核心规则和知识地图。
2. Standards、Workflows 和 Playbooks 仅在任务相关时加载。
3. 项目记忆只记录已验证事实、已接受决策、有效技术债和可复用经验。
4. 不按会话机械追加摘要；事实变化时更新，失效时删除或归档。

## 六、完成定义

任务完成需要满足与风险相称的条件：

- 用户要求的行为已实现或问题已回答。
- 相关验证已通过；无法执行的检查被明确说明。
- 没有未经说明的破坏性副作用或无关改动。
- 公共行为、命令、架构边界或长期决策发生变化时，相应文档已更新。
- 交付说明包含结果、验证和剩余风险。

发布、提交、打标签和部署是独立授权动作，不因代码完成而自动发生。
