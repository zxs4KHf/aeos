# 📑 事故复盘分析 (Postmortem) 模板

> [!NOTE]
> **使用指南**：当发生严重的系统崩溃、任务丢包、死锁锁死或线上发布失败等“事故”并解决后，智能体必须撰写事故复盘报告。复盘旨在客观还原事故过程，分析根本原因，并制定彻底根治的行动计划（Action Items），杜绝同类故障重复发生。本模板秉持“对事不对人（Blameless Postmortem）”原则。

## 三、 根本原因分析 (Root Cause Analysis - RCA)

> 深入到技术最底层分析“为什么”会发生这个问题。可使用 **“五个为什么 (5 Whys)”** 方法推进：

1. **为什么系统会无响应？**
   - 因为后台 Poller 被阻塞在了 SQLite 的写操作上。
2. **为什么 SQLite 写操作会被阻塞？**
   - 因为前一个 VS Code 会话的旧 Poller 进程没有退出，仍然占有着写排他锁。
3. **为什么旧 Poller 进程没有退出？**
   - 因为 VS Code 重启时，没有对沙箱内的 detached 外部后台进程发送退出信号。
4. **为什么没有对 Poller 进程做父子健康检测？**
   - 因为之前只设计了以文件锁状态来判定，未引入父进程存活心跳（`process.ppid`）的自愈逻辑。
5. **最终根本原因**：
   - 系统缺乏跨会话孤儿进程的自检测与自毁清理回收策略。
